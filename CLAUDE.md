## О проекте

GardenGram — интернет-магазин растений. Продаёт плодовые культуры (смородина,
земляника, яблони и др.) и декоративные растения (гортензии, лапчатка,
можжевельники и др.).

**Владельцы:** супружеская пара.
- Жена — эксперт по растениям, главный заказчик продукта, отвечает за каталог и ассортимент.
- Муж (Mikhail) — технический реализатор, работает с Claude Code.

**Локация бизнеса:** с. Иглино, Башкортостан. География продаж на старте — Россия.

**Цель:** запустить работающий магазин, наращивая функциональность маленькими
проверяемыми шагами.

## Как мы работаем

- Разработчик учится по ходу проекта — объясняй решения простым языком, не только давай код.
- Двигаемся **маленькими шагами**: одна задача → запуск → проверка → следующий шаг.
- Разработчик выступает «глазами и руками»: запускает код, проверяет результат, сообщает об ошибках.
- **Не строить лишнего раньше времени.** Закладывать в архитектуру только то, переделка
  чего потом дорога. Всё остальное — в backlog (см. ниже), реализуется после MVP.

## Технический стек

**Backend (основной фокus сейчас):**
- Python, Django 5 + Django REST Framework
- PostgreSQL (в dev — через Docker)
- Аутентификация: JWT (djangorestframework-simplejwt)
- Документация API: drf-spectacular (Swagger / OpenAPI)
- Зависимости уже заданы в `backend/requirements.txt`

**Инфраструктура (dev):**
- docker-compose поднимает PostgreSQL 16 и Redis 7
- Redis есть в инфраструктуре, но **очереди (Celery) на старте НЕ используем** —
  фоновые задачи (письма, вебхуки) пока обрабатываем синхронно. Добавим, когда реально упрёмся.

**Frontend / mobile — ПОЗЖЕ:**
- Сначала делаем и отлаживаем backend (данные и логику).
- Веб-интерфейс — следующий крупный этап.
- Мобильное приложение (React Native) — отдельная, более поздняя фаза.

## Текущее состояние кода (на момент написания)

Проект развивается на основе существующего каркаса (не с нуля). Что есть:

- `backend/apps/users/` — **единственное живое приложение**. Готовы регистрация,
  вход (JWT), endpoint `/me`. Используется стандартный Django User (см. решение ниже).
- `backend/apps/catalog/`, `backend/apps/core/`, `backend/common/` — **пустые заготовки**.
- `backend/config/` — настройки Django (settings, urls, wsgi, asgi).
- Готовые маршруты: `auth/login`, `auth/refresh`, `auth/register`, `me`,
  `api/schema`, `api/docs`, `admin`.
- **Миграции не создавались и не применялись. База пустая.** Это важно — см. решение о пользователе.

**Незавершённое / требует внимания:**
- Каталог товаров (главная часть магазина) ещё не существует.
- Нет сервиса backend в docker-compose — запуск Django пока вручную.
- Нет `.env` (правильно, что не в репозитории), но нужен `.env.example` как шаблон.
- В папке `exchange/` лежит случайно попавший туда личный документ — убрать из репозитория
  (репозиторий публичный).

## Ключевые архитектурные решения

Эти решения приняты осознанно. Не менять без обсуждения с разработчиком.

1. **Кастомная модель пользователя — сделать ДО первой миграции.**
   Бизнесу нужны дополнительные поля у пользователя: телефон (формат +7), город, адрес доставки.
   Так как база ещё пустая и миграций нет — это идеальный момент ввести кастомную модель
   пользователя. После наполнения базы менять модель пользователя в Django крайне болезненно.

2. **Задел под нескольких продавцов — без построения системы продавцов.**
   Сейчас продавец один (владелец). В будущем возможны другие продавцы/питомники.
   Модель товара должна быть спроектирована нейтрально, чтобы связь с продавцом можно
   было добавить позже без болезненной переделки. Но саму систему мультипродавца
   (кабинеты, разделение товаров, выплаты) сейчас НЕ строим. Все товары пока принадлежат владельцу.

3. **Каталог — дерево категорий.**
   Категории древовидные, с делением на плодовые / декоративные и далее по видам растений
   (например: хвойные → можжевельник; плодовые → смородина). Референсные сайты подтверждают
   потребность в богатой навигации по видам.

4. **Значения, которые владельцы меняют сами, живут в админке, а не в коде.**
   Цены, остатки, пороги, тексты и подобное должны редактироваться через Django admin
   без участия программиста. Ничего из этого не хардкодить.

## Доставка

Способы (заложить как данные; конкретику допиливаем поэтапно):
- **Самовывоз** — основной способ.
- **Локальная доставка по с. Иглино** силами владельца (сам везёт на машине).
  Бесплатна при заказе от определённой суммы. **Порог суммы — настройка в админке**,
  владельцы меняют её сами в зависимости от ситуации (экономика, загрузка).
  При оформлении заказа покупателю показывается подсказка о доступности/условиях бесплатной доставки.
- **Доставка по России** (Почта / СДЭК и т.п.) — будущая фаза, пока не реализуем.

## Границы MVP

**Ядро MVP (делаем по очереди):**
каталог (дерево категорий, товары, варианты/цены/остатки) → корзина → оформление заказа
→ оплата.

**Оплату** прорабатываем через Hosted Checkout провайдера, но конкретную интеграцию
откладываем — сначала каталог, корзина и заказ. На этапе модели заказа закладываем
поле статуса оплаты, саму оплату подключаем позже.

**Модель заказа** проектируем с прицелом на будущие фичи (см. backlog): у заказа есть
статус (например: новый / собран / выдан), место под желаемое время приезда покупателя,
уникальный идентификатор заказа. Сами фичи поверх этого — потом.

## Backlog (будущие фичи — НЕ реализуем сейчас, но не теряем)

Записано, чтобы не забыть. Реализуется после рабочего MVP, по приоритету.

1. **Скидки и акции.** Настраиваются из админки: процент скидки, название акции,
   перечень товаров в акции; несколько акций могут действовать одновременно.
   Примеры: скидка учителям в День учителя, скидка финишёрам Уфимского марафона.

2. **Уведомления владельцам о новом заказе.** При оформлении заказа владельцам
   приходит уведомление: кто заказал, перечень товаров, оплачен ли заказ.
   Желательно — указанное покупателем время приезда, чтобы собрать заказ заранее.
   (Базовое уведомление — раньше в очереди, чем пункты ниже.)

3. **Выдача заказа по идентификатору (как пункт выдачи маркетплейса).**
   Покупатель показывает на телефоне QR/код, владелец считывает и сразу видит заказ,
   выносит и отдаёт. (Отдельный модуль, дальше в очереди.)

4. **Партнёрские коллаборации.** За покупку покупатель получает ваучер / QR-код
   на скидку или услугу у партнёра. (Отдельный модуль, дальше в очереди.)

5. **Прочее из исходных планов (вторая фаза):** отзывы/рейтинги, избранное,
   интеграция доставки (СДЭК/Почта), отдельный веб-фронтенд, мобильное приложение iOS.

## Чего НЕ делать

- Не хардкодить значения, которые владельцы должны менять сами (цены, пороги, тексты) — они в админке.
- Не строить систему мультипродавца сейчас (только нейтральный задел в модели).
- Не реализовывать пункты backlog, пока не готов базовый MVP.
- Не подключать Celery/очереди, пока нет реальной необходимости.
- Не уходить во фронтенд/мобильное приложение, пока не отлажен backend.
- Не менять кастомную модель пользователя «задним числом» после наполнения базы.
