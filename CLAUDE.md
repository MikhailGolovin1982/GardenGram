## О проекте

GardenGram — интернет-магазин растений. Продаёт плодовые культуры (смородина,
земляника, яблони и др.), декоративные растения (гортензии, лапчатка, можжевельники
и др.) и сопутствующие товары (горшки, грунты, торф, удобрения, препараты).

**Владельцы:** супружеская пара.
- Жена (Людмила) — эксперт по растениям, главный заказчик продукта, отвечает за каталог и ассортимент.
- Муж (Mikhail) — технический реализатор, работает с Claude Code.

**Локация бизнеса:** с. Иглино, Башкортостан. География продаж на старте — Россия.

**Цель:** запустить работающий магазин, наращивая функциональность маленькими
проверяемыми шагами.

- Разработчик учится по ходу проекта — объясняй решения простым языком, не только давай код.
- Двигаемся **маленькими шагами**: одна задача → запуск → проверка → следующий шаг.

## Технический стек

**Backend (основной фокус сейчас):**
- Python, Django 5 + Django REST Framework
- PostgreSQL (в dev — через Docker)
- Аутентификация: JWT (djangorestframework-simplejwt)
- Документация API: drf-spectacular (Swagger / OpenAPI)
- Зависимости заданы в `backend/requirements.txt`

**Инфраструктура (dev):**
- docker-compose поднимает PostgreSQL 16 и Redis 7
- Redis есть в инфраструктуре, но **очереди (Celery) на старте НЕ используем** —
  фоновые задачи пока обрабатываем синхронно. Добавим, когда реально упрёмся.
- Локальные настройки (имя БД, пользователь, пароль) — в файле `.env` (НЕ в git).
  Шаблон — `.env.example` (в git). В settings.py для Codespaces автоматически
  разрешён forwarded-домен (ALLOWED_HOSTS + CSRF_TRUSTED_ORIGINS через CODESPACE_NAME).

**Frontend / mobile — ПОЗЖЕ:**
- Сначала делаем и отлаживаем backend (данные и логику).
- Веб-интерфейс — следующий крупный этап. Мобильное приложение (React Native) — более поздняя фаза.

## КАТАЛОГ (РЕАЛИЗОВАНО)

Каталог — сердце магазина. Решения ниже согласованы с Людмилой (экспертом по растениям).
Модели, админка и read-only API готовы и проверены; решения раздела закреплены в коде.

### Два независимых механизма

Важно различать две вещи:
1. **Дерево категорий** отвечает на вопрос «ЧТО это и ГДЕ лежит» (гортензия — в декоративных;
   инсектицид — в защите растений).
2. **Варианты товара** отвечают на вопрос «в каком ВИДЕ продаётся и ПОЧЁМ»
   (гортензия в горшке 5 л за столько-то; торф в мешке 50 л за столько-то).

### Дерево категорий

- **Единое дерево** категорий на весь каталог, с **произвольной вложенностью**.
- Ветки верхнего уровня независимы и НЕ смешиваются (растения отдельно, удобрения отдельно).
- Людмила сама строит и углубляет ветки через админку, без программиста.
- Реальный пример структуры (как образец):
  - Декоративные растения → гортензии, дерены, туи, лапчатки, спиреи, жасмины, можжевельники…
    - (с углублением где нужно: Гортензии → метельчатые / крупнолистные → сорта)
  - Ягоды → земляника, смородина, жимолость, крыжовник…
  - Деревья → яблони, груши, сливы, вишня…
  - Удобрения и препараты → Препараты для защиты (инсектициды, пестициды, биопрепараты)
    / Препараты для подкормки (калийные, фосфорные…)
  - Горшки
  - Грунты и торф
- Ботанические группы (метельчатые/крупнолистные и т.п.) — это просто подкатегории дерева,
  отдельный механизм «свойств» для этого НЕ нужен.

### Товар (карточка растения / сопутствующего товара)

Поля карточки:
- **Название на русском** — обязательное.
- **Сорт** — необязательное отдельное поле (нужно для будущей фильтрации; есть не у всех позиций).
- **Латинское название** — необязательное. Правило отображения: если заполнено →
  показывать «Русское название (Латинское название)»; если пусто → только русское.
- **Описание** — обязательное.
- **Фото** — необязательное, несколько штук (5-6). Грузятся по мере возможности.
- **Категория** — место в дереве.

### Варианты товара (у каждого своя цена и остаток)

Один вид/сорт = одна карточка. Внутри — варианты, отличающиеся тем, КАК товар продаётся.

- Для **растений** вариант определяют:
  - тип корневой системы: ОКС (без горшка) / ЗКС (в контейнере);
  - объём горшка для ЗКС (от 2-3 л до 20-40 л);
  - возраст — НЕ жёсткая характеристика, необязательное справочное поле-комментарий
    (растения живые, развиваются по-разному; на цену влияет слабо).
- Для **сопутствующих товаров** (торф, грунт, горшки, удобрения) вариант — это
  объём/размер фасовки (мешок 50 л, горшок такого-то размера). Корневой системы у них нет.
- **ВАЖНО (нейтральный задел):** модель товара и варианта должна быть достаточно ОБЩЕЙ,
  чтобы описывать и растения (ОКС/ЗКС + объём), и сопутствующие товары (объём/размер фасовки),
  НЕ навязывая «растительные» поля всем подряд. Это дешёвый задел сейчас, дорогая переделка потом.

Цена и остаток — на уровне ВАРИАНТА, не товара:
- **Цена** — базовая, всегда за 1 штуку. (Оптовые/индивидуальные скидки — backlog.)
- **Остаток** — гибкий: поштучный учёт необязателен. Владелец управляет наличием через статус;
  число количества опционально (точное число известно не всегда). Жёсткий склад — backlog.
- При отсутствии товар НЕ скрывается, а показывается со статусом **«Нет в наличии»**
  (важно для сезонных растений — покупатель видит ассортимент).

## Бэкап данных каталога

**Зачем.** База живёт внутри Codespace и теряется при его пересоздании. Код в git, а
данные — нет. Поэтому данные каталога выгружаем в фикстуру и коммитим в репозиторий.

**Как сделать бэкап** (из папки `backend/`, где лежит `manage.py`):
```
python manage.py dumpdata catalog --indent 2 --output fixtures/catalog.json
```
затем закоммитить файл `backend/fixtures/catalog.json`.

**Как восстановить** (из папки `backend/`):
```
docker compose up -d db
python manage.py migrate
python manage.py loaddata fixtures/catalog.json
```

**Правило.** Делать бэкап и коммитить его **после каждой существенной сессии наполнения
каталога** (добавили/поправили товары — обновили фикстуру).

**Ограничение / TODO.** Фото (`media/`) в этот бэкап **НЕ попадают** — папка в `.gitignore`,
фикстура хранит только пути к файлам. Бэкап картинок — отдельная задача на будущее.

**users НЕ выгружаем** в этот бэкап: там пароли и телефоны, а репозиторий публичный.
Суперпользователь восстанавливается командой `createsuperuser`.

## КОРЗИНА (РЕАЛИЗОВАНО)

Приложение `backend/apps/cart/`. Решения зафиксированы в коде и проверены end-to-end.

- **Гостевая + пользовательская.** Класть товары можно без регистрации. Гость опознаётся
  анонимным токеном (UUID), который клиент хранит и шлёт в заголовке **`X-Cart-Token`**;
  залогиненный — по аккаунту. Одна корзина на субъекта (uniq по `user` / по `token`).
  Сессии/cookie сознательно НЕ используем (проект API-first на JWT).
- **Слияние при входе.** Отдельный `POST /api/v1/cart/merge/`: фронт после логина шлёт
  гостевой токен, гостевая корзина вливается в аккаунт. Совпавшие варианты — количества
  **суммируются**, дублей нет; если у юзера корзины не было — гостевая «усыновляется».
- **Единица корзины — `ProductVariant`** (не товар) + количество.
- **Цена живая** — берётся из варианта на момент показа (заморозка цены будет в заказе).
- **Наличие:** добавлять можно только доступные варианты (активные + `IN_STOCK`). Если
  лежащий в корзине вариант стал недоступен — он ОСТАЁТСЯ, помечается `is_available_now=false`
  и в сумму НЕ входит (покупатель не обязан удалять вручную).
- **Количество наличием НЕ ограничиваем** (точного учёта нет; заказы подтверждает владелец).
- **Сумма** считается по активным позициям (цена × кол-во); доставка в корзине НЕ считается.
- **API:** `GET/DELETE /api/v1/cart/` (показать/очистить), `POST /api/v1/cart/items/`
  (добавить), `PATCH/DELETE /api/v1/cart/items/{id}/` (кол-во/удалить),
  `POST /api/v1/cart/merge/`. Всё доступно гостю (`AllowAny`), merge — только залогиненному.
- Есть unit-тесты на слияние и расчёт суммы; админка корзины — для поддержки/отладки.

## Границы MVP

**Ядро MVP (по очереди):**
каталог (дерево категорий, товары, варианты/цены/остатки) → корзина → оформление заказа → оплата.

- Пользователь, каталог, корзина — ГОТОВЫ. Сейчас — **оформление заказа**.
- **Оплата** через Hosted Checkout, интеграцию откладываем. В модели заказа заложим поле статуса оплаты.
- **Модель заказа** проектируем с прицелом на будущее: статус (новый / собран / выдан),
  место под желаемое время приезда покупателя, уникальный идентификатор заказа.

## Backlog (будущие фичи — НЕ сейчас, но не теряем)

1. **Скидки, акции и оптовые/индивидуальные цены.** Настраиваются из админки: процент,
   название акции, перечень товаров; несколько акций одновременно. Оптовая цена при заказе
   от суммы (напр. от 20 000 ₽ — гортензия 350 ₽ вместо 500 ₽). Примеры акций: учителям в
   День учителя, финишёрам Уфимского марафона.
2. **Уведомления владельцам о новом заказе** (кто, что, оплачено ли, желаемое время приезда).
3. **Выдача заказа по идентификатору** (QR/код на телефоне, как пункт выдачи маркетплейса).
4. **Партнёрские коллаборации** (ваучер/QR за покупку на услугу/скидку у партнёра).
5. **Расширение входа:** вход по email; вход через российские соцсети и Google
   (зарубежные сервисы в РФ нестабильны); привязка нескольких контактов к одному аккаунту;
   SMS-подтверждение телефона.
6. **Жёсткий складской учёт** остатков (точный поштучный контроль).
7. **Фильтры по свойствам** товара (по сорту, по характеристикам) сверх дерева категорий.
8. **Фото на уровне варианта.** Сейчас фото привязаны к товару (общие для всех вариантов).
   В будущем — возможность прикреплять разные фото к конкретным вариантам (ОКС / ЗКС /
   горшки разного объёма выглядят по-разному). Добавляется аддитивно поверх существующей
   модели ProductImage (вторая необязательная ссылка на вариант или отдельная модель),
   без переделки. Делать при реальной необходимости после наполнения каталога.
9. **Улучшения админки каталога для удобства наполнения** (несложные, делать отдельным заходом):
   (а) в списке товаров показывать количество/сводку вариантов, чтобы не проваливаться внутрь
   карточки ради этого;
   (б) у вариантов внутри карточки — понятные заголовки (напр. «ОКС — 600 ₽» / «ЗКС 5 л — 800 ₽»)
   и более чёткое визуальное разграничение блоков вариантов, чтобы не путаться, где какой.
10. **Прочее (вторая фаза):** отзывы/рейтинги, избранное, сравнение товаров, интеграция
   доставки (СДЭК/Почта), статьи/блог о растениях, скачиваемые прайс-листы, оптовый раздел,
   региональность, отдельный веб-фронтенд, мобильное приложение iOS.

## Чего НЕ делать

- Не хардкодить значения, которые владельцы меняют сами (цены, пороги, тексты) — они в админке.
- Не строить систему мультипродавца сейчас (только нейтральный задел в модели).
- Не реализовывать пункты backlog, пока не готов базовый MVP.
- Не подключать Celery/очереди, пока нет реальной необходимости.
- Не делать жёсткие «растительные» поля обязательными для всех товаров (есть сопутствующие).
- Не уходить во фронтенд/мобильное приложение, пока не отлажен backend.
- Не менять кастомную модель пользователя «задним числом» после наполнения базы.
- Не вводить SMS-подтверждение и уникальность email на старте.
