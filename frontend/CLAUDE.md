## Что это и зачем

Фронтенд GardenGram — сайт интернет-магазина растений, витрина для ПОКУПАТЕЛЯ.
Строится ПОВЕРХ готового REST API бэкенда (Django). Бэкенд — «двигатель данных»,
сайт — «кузов и салон»: получает данные от API и показывает покупателю.

**Важно:** бэкенд (каталог, корзина, заказы, аутентификация) УЖЕ ГОТОВ и проверен.
Сайт ничего не считает «сам» — вся бизнес-логика на бэкенде. Сайт только показывает
данные и отправляет действия пользователя в API.

**Управление магазином (каталог, цены, остатки)** — только через Django-админку.
На сайте админских функций НЕТ. Сайт = чистая витрина для покупателя.

## Технический стек

- **React** + **Vite** (сборщик и dev-сервер).
- Чистый клиент к REST API. Данные отдаёт Django, рендеринг на стороне браузера.
- Next.js НЕ используем (серверный рендеринг сейчас не нужен — данные у Django).
- Управление состоянием и роутинг — добавляются по мере необходимости, не заранее.

**Почему React (а не Vue/шаблоны):** в планах два мобильных приложения (для покупателя
и для владельца) — React даёт переиспользуемый навык через React Native. Это инвестиция
с прицелом на мобильную фазу, а не разовая трата.

## Структура проекта (монорепо)

Фронтенд живёт в папке `frontend/` рядом с `backend/` в ОДНОМ репозитории.

```
GardenGram/
├── CLAUDE.md            ← корневой: проект в целом + бэкенд
├── backend/             ← Django (готов)
└── frontend/            ← React + Vite (этот файл здесь)
    └── CLAUDE.md        ← память фронтенда (этот файл)
```

**Два сервера в одном Codespace, в двух терминалах:**
- Django (бэкенд, API) — порт **8000**
- Vite (фронтенд, сайт) — порт **5173**

Каждый порт в Codespaces = свой публичный forwarded-адрес. Адрес Django, по которому
сайт ходит за данными, динамический (зависит от имени Codespace) — задаётся через
переменную окружения (Vite: `VITE_*`), НЕ хардкодить.

**Осторожно с терминалами:** у разработчика `Ctrl+C` в терминале прерывает процесс
Claude Code. Серверы (Django, Vite) и Claude Code держим в РАЗНЫХ терминалах, чтобы
остановка одного не убивала другое.

## API, на которое опирается сайт (контракт с бэкендом)

Это мост между фронтендом и бэкендом. Сайт использует эти эндпоинты. Если API
меняется — обновлять этот блок. Детали логики каждого — в корневом CLAUDE.md.

**Каталог (read-only):**
- `GET /api/v1/catalog/categories/` — дерево категорий.
- `GET /api/v1/catalog/products/` — список товаров (фильтр по категории, поиск, пагинация).
- `GET /api/v1/catalog/products/{id}/` — карточка товара с вариантами и ценами.

**Корзина** (гость — по заголовку `X-Cart-Token`, юзер — по JWT):
- `GET /api/v1/cart/` — показать корзину.
- `DELETE /api/v1/cart/` — очистить.
- `POST /api/v1/cart/items/` — добавить вариант.
- `PATCH /api/v1/cart/items/{id}/` — изменить количество.
- `DELETE /api/v1/cart/items/{id}/` — удалить позицию.
- `POST /api/v1/cart/merge/` — слить гостевую корзину в аккаунт (только залогиненному).

**Заказы:**
- `POST /api/v1/orders/` — оформить заказ (гость + юзер).
- `GET /api/v1/orders/preview/` — суммы + опции доставки + подсказка про порог.
- `GET /api/v1/orders/{access_token}/` — просмотр заказа гостя по секретному токену.
- `GET /api/v1/orders/` — свои заказы (только залогиненному).

**Аутентификация (JWT, вход по телефону):**
- `POST /api/v1/auth/login/` — вход.
- `POST /api/v1/auth/register/` — регистрация.
- `GET /api/v1/auth/me/` — текущий пользователь.

> ВАЖНО: точные пути сверены с бэкендом по Swagger. При расхождении — Swagger
> (`/api/docs/`) источник правды, поправить здесь.

## Ключевые особенности, которые сайт обязан учитывать

Это поведение бэкенда, которое влияет на интерфейс. Не игнорировать.

- **Гостевой поток — основной.** Корзину и заказ можно пройти БЕЗ регистрации:
  гость опознаётся токеном `X-Cart-Token` (UUID), который сайт хранит у себя и шлёт
  в каждом запросе к корзине. Вход/регистрация — необязательны для покупки.
- **Цена и наличие — на уровне ВАРИАНТА**, не товара. У товара несколько вариантов
  (ОКС/ЗКС, объёмы), у каждого своя цена и статус. Единица корзины — вариант, не товар.
- **«Нет в наличии» — показывать, НЕ скрывать.** Недоступный товар/вариант виден
  со статусом «Нет в наличии» (важно для сезонных растений — покупатель видит ассортимент).
- **Недоступная позиция в корзине ОСТАЁТСЯ.** Если лежащий в корзине вариант стал
  недоступен — он не удаляется, помечается флагом (`is_available_now=false`) и НЕ входит
  в сумму. Сайт показывает такую позицию визуально отдельно, не молча выкидывает.
- **Латинское название** показывать по правилу: если заполнено → «Русское (Латинское)»;
  если пусто → только русское.
- **Доставка** считается на бэкенде (`/orders/preview/`): самовывоз = 0; доставка по
  Иглино — платная, но бесплатна от суммы-порога. Сайт показывает то, что вернул API,
  сам НЕ считает.

## Экраны на старте (MVP сайта)

Минимум, при котором покупатель проходит полный путь: каталог → карточка → корзина → заказ.

1. **Главная** — точка входа: логотип, навигация по категориям, немного товаров.
2. **Каталог** — дерево категорий + сетка карточек, поиск, фильтр по категории.
   Сердце сайта (референсы ценятся за удобный каталог). `categories/` + `products/`.
3. **Карточка товара** — фото, название, описание, варианты с ценами, «в корзину».
   `products/{id}/`.
4. **Корзина** — список, изменение количества, удаление, сумма. `cart/`.
5. **Оформление заказа (checkout)** — форма покупателя, выбор доставки, итог.
   Гость + юзер. `orders/preview/` + `orders/`.
6. **Страница заказа** — подтверждение (номер `GG-...`), просмотр гостя по токену.
   `orders/{access_token}/`.
7. **О нас** — статичная страница. Лицо магазина: Людмила — кандидат биологических
   наук, специализация на микроклональном размножении растений. Сигнал доверия и
   отстройка от конкурентов. Просто объяснить покупателю выгоду (здоровые сортовые
   растения, научный подход). Семейное дело, Иглино, Башкортостан. Живое фото.
   Технически простая; контент (текст/фото) готовится отдельно, не спеша.
8. **Вход / регистрация** — `auth/login/`, `auth/register/`. ТОЛЬКО если успеваем
   к дедлайну; иначе — backlog первой задачей. Весь путь покупки работает без неё.

## Порядок работы (шаги реализации)

Принцип: сначала каркас и связь, потом данные, потом действия. Красоту наводим
поверх, когда костяк держится.

- **Шаг 0** — фундамент: создать `frontend/` через Vite, запустить пустой React,
  настроить адрес API, ПРОВЕРИТЬ связь (фронт дёргает `categories/`, показывает сырой
  список). Доказывает, что вся связка работает (Vite ↔ Django ↔ forwarded ↔ CORS).
- **Шаг 1** — каркас навигации: роутинг + пустые заглушки всех экранов.
- **Шаг 2** — каталог (читающая часть): список + дерево категорий + поиск.
- **Шаг 3** — карточка товара (варианты, цены). Завершает «смотрящую» часть.
- **Шаг 4** — корзина (первая «пишущая» логика: токен гостя, сумма).
- **Шаг 5** — checkout + страница заказа. Замыкает путь покупки.
- **Шаг 6** — «О нас» (статика, можно параллельно).
- **Шаг 7** — вход/регистрация (если остаётся время).

После Шага 3 — смотрибельная витрина (можно показать Людмиле). После Шага 5 —
работающий магазин.

## Чего НЕ делать на фронтенде

- Не дублировать бизнес-логику бэкенда (цены, доставку, наличие считает API).
- Не делать админских функций на сайте (управление — в Django-админке).
- Не хардкодить адрес API (переменная окружения).
- Не строить личный кабинет с историей заказов сейчас (backlog).
- Не строить блог/статьи сейчас (backlog).
- Не подключать Next.js / серверный рендеринг (не нужно при API на Django).
- Не уходить в шлифовку дизайна, пока не работает базовый путь покупки.
- Не скрывать товары «нет в наличии» — показывать со статусом.
